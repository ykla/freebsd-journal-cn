# 读者来信

作者：**Michael W. Lucas**（<letters@freebsdjournal.org>）

嗨 Michael，

我们在为 FreeBSD 期刊头脑风暴专栏点子，而 Kode Vicious 建议你可能愿意为我们处理“读者来信”专栏。人们会向期刊提交他们的问题，然后你来为我们回答。你有兴趣吗？

嗨 George，

这是个糟糕的主意。简直糟糕透顶。这是互联网时代。没人读读者来信专栏、建议专栏或类似的东西。我们有 Stack Exchange 和各种让人求助的地方。

FreeBSD 有很多地方可以去寻求具体帮助。系统自带帮助，在 man 页面中。许多类 Unix 操作系统做出了荒谬的决定，将 man 页面单独捆绑，FreeBSD 随系统附带手册。实际上，你无法不安装手册。当然，你可以构建一个不包含手册的 FreeBSD，但要做到这一点需要阅读大量 man 页面。

人们说手册不是教程，他们没错。这就是 FreeBSD 有 Handbook 和大量文章的原因。与 man 页面不同，你可以选择不在 FreeBSD 主机上安装它们。你可以在线浏览所有文档，地址是 <https://docs.freebsd.org>。

新用户可以从 FAQ（Frequently Asked Questions）文件开始，其中包含几十个问题和答案。它涵盖从硬件兼容性到 ZFS 的所有内容，虽然“The FreeBSD Funnies”中的一些段子已经严重过时——如今内存中不再有什么划痕了，1996 年就修复了那个 bug——文档的其余部分坚如磐石。现在看来，我意识到它对新用户多么有用。我仍记得当我理解为什么 **du(1)** 和 **df(1)** 对磁盘空间使用给出不同答案时的那种顿悟感。花一个小时阅读 FAQ 会给新用户带来一次又一次的顿悟。

然后是 Handbook。它按任务分解。如果用户的提问比 FAQ 稍微深入一些，Handbook 就能帮到你。一些材料为读者定位，对新 FreeBSD 管理员来说非常值得一读，让他们理解为什么 **/usr/local** 有那么多内容，而其他系统把所有东西都塞到 **/etc** 和 **/bin** 里。

此外，现在有各种 FreeBSD 相关站点。连我的博客都有一些 FreeBSD 教程。如果有人真的写信求助，那是因为他们没用这些资源。

——ml

George，

好吧，让我们谈谈那些确实有 Handbook 中没有的问题的人。当我开始使用 FreeBSD 时，你可以通过 <FreeBSD-questions@FreeBSD.org> 邮件列表获得帮助。它现在还在。邮件列表上的人想回答问题。他们专门订阅是为了帮助人们解决问题。这些勇敢的人自愿花时间回答用户的问题。我能做那些英雄做不到的什么？

对于那些已经忘记电子邮件如何工作的年轻人，FreeBSD 论坛在 <https://forums.freebsd.org>。与邮件列表不同，论坛按类别分解。用户可以深入讨论安装、存储、硬件、包或任何内容。每当我看论坛，都会发现有趣的讨论。

邮件列表档案里有四分之一个世纪的问题解决。我能说什么没被反复说过多次的话？

这些渠道非常适合特定硬件问题。Handbook 和 FAQ 是 FreeBSD 生态系统中的永久固定设施——它们已经存在了几十年。但是，如果你的廉价新笔记本电脑中的某个芯片组给你带来麻烦，你可以搜索邮件列表或论坛看看是否有人对该硬件有同样的问题。

不愿在邮件列表档案或搜索论坛上用 DuckDuckGo 的用户肯定也不会费心给我写一封连贯的信。

——ml

该死，George，你就是不放弃，是吧？

好吧，好吧。我们来走一遍。用户有一个问题。一个真正独特的问题，邮件列表档案、论坛中都找不到。互联网上唯一模糊提到这个问题的参考在一个 darknet 站点上，以西伯利亚语写成。他们确实真诚地遇到了麻烦。

在任何人能帮助这个用户之前，他们必须有用地描述他们的问题。这意味着他们必须发送完整的问题描述。大多数撰写求助请求的人都不会费心提供 `uname -a` 的输出和 `dmesg.boot` 的副本。他们不会麻烦自己提供实际错误输出或 **/var/log/messages** 的内容。或者他们“有用”地剥离了他们认为不相关的内容，比如在他们的 Web 服务器日志中贯穿始终的“PHP 正在转储核心”消息。

还有一件事。人们希望得到与 FreeBSD 无关的帮助。他们知道这与 FreeBSD 无关。然而他们却向 FreeBSD 邮件列表发消息？我是说，这太粗鲁了。

说到粗鲁——人们求助时礼貌一点会死吗？邮件列表或论坛上任何花时间帮助用户的人都是自愿的。他们有更好的事情可做，而不是忍受你的脾气。我的意思是，我理解计算机真的能让人抓狂。我自己也不止一次站在屋顶上对缓冲区缓存大喊下流脏话——谁没这么干过？但没必要把气撒在试图帮助你的人身上。

大多数情况下，仅仅是撰写问题描述就足以让我自己的大脑解决问题。

而除了高压电，没有什么能鼓励人们买我的书。

所以，不。让有困难的用户去邮件列表或论坛。我有足够的事要做。

——ml
