# 书评：《凤凰项目：一个 IT 运维的传奇故事》

作者：**Benedict Reuschling**

《凤凰项目：一个 IT 运维的传奇故事》

- **作者：** Gene Kim、Kevin Behr、George Spafford
- **出版社：** IT Revolution Press（第 3 版，2018 年）
- **定价：** 24 美元
- **ISBN：** 9781942788294
- **页数：** 432 页

《凤凰项目：一个 IT 运维的传奇故事》是一部以 IT 世界为背景的小说，由 Gene Kim、Kevin Behr 与 George Spafford 合著。虽然书中的人物和公司都是虚构的，但所描绘的情境与人物关系却让人倍感真实。故事开篇，主角 Bill Palmer 在一堆前任留下的烂摊子中升任 IT 运营副总裁。

这本书出色地描绘了几乎任何公司都可能遇到的问题。读者能够联想到自己经历过的类似情境以及这些问题对业务的影响。我很有兴趣看看从最初几章的一片混乱中如何收场。此外，我在自己的多份工作中也看到大量类似的经历，无论是处理方式，还是不处理的方式，都极为相似。有时候，并不一定是不知道更好的做法——正如书中所描绘的那样——人们只是看不到其他做事的方式。我认为这正是 DevOps 的有趣之处，也是它带来的承诺：让 IT 工作以及与其他部门的协作变得更好。DevOps 最终会改变日常做事的方式。

公司常常把 IT 视作纯粹的成本中心，或是不创造多少价值的“必要之恶”。我记得一位教授曾提醒我们：IT 不是我们的核心业务，但它是我们业务的核心。这本书描述了 IT 起初如何被视作其他所有部门都要依赖、却不断制造麻烦与困扰的部门。随着故事推进，IT 人员逐渐意识到需要跳出本部门的边界去审视问题。识别销售、财务等其他部门需要 IT 提供什么，会改变优先级和目标。对这些部门产生直接影响，会以积极的方式改变业务成果，远不止可靠地运行服务器和对生产系统应用变更。这就是 DevOps 的承诺之一：随着时间推移，它最终会改造整个公司，使其成为更好、更高效、更具生产力的组织。

批评者可能会认为书中描述的一些方法在大公司中实施过于天真，或者结果不可能像小说中那样迅速显现。然而，书中也描写了抗拒 Bill 实施计划的人，他们试图破坏他的努力。我喜欢作者用心地把一些（虚构的）邮件以你在邮件程序中看到的形式记录下来（包括 From:、To: 和 Subject: 行）。这使书更真实、更具商务感。此外，关于等待时间是资源忙碌时间除以空闲时间之比的图表让我尤其有启发。它解释了很多事情，也提醒你小心不要过度承诺自己。

我特别喜欢的一点是，这本书不仅关注 IT 运营的业务层面，还描写了它如何影响 Bill 与妻子和孩子的个人生活。Bill 经常在下班后在家接到电话，需要处理眼前的业务问题，而不是陪伴家人。IT 行业出问题时，往往牵动整个家庭。

阅读这本书是我大学敏捷软件开发课程学习的延续。The Phoenix Project 中提及了丰田生产系统、精益制造和看板等方法，并将其应用于具体的商业场景。虽然我希望作者能更详细地阐述看板及其使用方法，但我也理解这只是一个工具，是 DevOps 大局中（虽然重要的）达到目的的手段。市面上还有其他关于看板的书籍，这本书激发了我尝试将其用于个人项目的兴趣。

并非所有读者都处在 Bill 那样的管理职位上，无法一次性改造整个部门。但这些概念对个人同样适用——比如识别并优先处理最重要的工作，通过不断应用和改进来精进技能，绘制价值流图并使用看板来可视化工作，消除浪费和返工。当然，其中一些概念需要更深入的阐述和实施技巧，这正是附录中提供资源指南的原因。此外，作者还提到了 The DevOps Cookbook，现更名为《DevOps 实践指南》，建议在阅读《凤凰项目：一个 IT 运维的传奇故事》之后阅读。

总而言之，我愿意向所有在 IT 行业工作的人推荐这本书。故事足够有趣，非 IT 行业的读者也能借此了解 IT 部门在应对什么。书中关于工作的理念几乎可以应用于任何职业。•

**BENEDICT REUSCHLING** 于 2009 年加入 FreeBSD 项目。2010 年获得完整的文档提交权限后，他积极开始指导他人成为 FreeBSD committer。他是 BSD 认证小组的监考官，于 2015 年加入 FreeBSD 基金会，目前担任副总裁。Benedict 拥有计算机科学理学硕士学位，在德国达姆施塔特应用技术大学教授面向软件开发者的 UNIX 课程。
