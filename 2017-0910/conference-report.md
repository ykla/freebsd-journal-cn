# 会议报告

- 原文：[BSDCam 2017 Trip Report](https://freebsdfoundation.org/wp-content/uploads/2017/10/Conference-Report.pdf)
- 作者：MICHAEL W LUCAS

数十年来，FreeBSD 的开发与协调从纯线上协作，逐渐转向越来越多地依赖面对面的协调与合作。FreeBSD 基金会在 BSDCan、EuroBSDCon 和 AsiaBSDCon 之前都会赞助一场开发者峰会，让前去参会的开发者能顺便利用机票解决一些问题。互联网确实便于协调，但比不上一群开发者花 10 分钟在白板上勾画，把某件事做到万无一失。除了协调工作之外，会议开发者峰会都是有层级结构的。议程固定，议题事先确定。每场会议都有主持人。会议可能极具信息量，也可能演变成激烈的争论，或介于两者之间。

BSDCam……有点不一样。它是 FreeBSD 生态系统中不可或缺的一部分。不过，我通常不会参加。但现在不是平常时候。我正在写《Absolute FreeBSD》的新版。让我惊讶的是，人们在规划部署和运营时竟然会依赖这本书。这让我感到欣慰，但也增加了把事情写对的压力。2000 年我写第一本 FreeBSD 书时，十几个邮件列表提供了 FreeBSD 开发的权威信息，一个人能读完所有这些列表。如今已不可能——而且邮件列表只是 FreeBSD 社交体系的一个狭窄方面。别误会——了解大家在做什么、系统如何运作并不难。但要了解大家将要做什么、系统将如何运作就没那么容易了。如果这本书要经得起未来考验，我必须离开舒适的小窝，踏入英国剑桥的“荒野”。可悲的是，BSDCam 主席认同了我的逻辑，于是我登上一个铝制死亡陷阱——抱歉，是“商用客机”——从底特律被甩到希思罗。一个周三早上，我到了剑桥大学的 William Gates 大楼，多亏了源源不断的正宗英式茶，意识才勉强钉在身上。

BSDCam 仅限受邀参加，场地也只能容纳 50 人左右。你必须正在积极参与 FreeBSD 工作才能拿到邀请。开发者从世界各地赶来。然而，会议没有议程。Robert Watson 是主席，但议题不由他决定。他绕场一周，让每个人自我介绍，说说自己在做什么，并声明想在会议期间讨论什么。大家感兴趣的议题被汇总。最受欢迎的议题分配到时间段和两个大房间之一。对小众议题感兴趣的人则被邀请使用小型分组会议室。然后真正的乐趣开始了。

我一开始旁听了虚拟化研讨会。整整两小时，大家讨论 FreeBSD 虚拟化的需求、优势和弱点。哪里需要改进？接口应该是什么样？哪些兼容性重要，哪些不重要？会议结束时，二十来号人达成了合理共识，更重要的是，有些人在自己的待办清单上加了几项。这样的情形在十几个议题上重复。我对安全缓解技术、FreeBSD 的云支持、TCP/IP 改进、FreeBSD 教学进展等领域的现状有了清晰把握。BSDCan 开发者峰会上关于打包基本系统的演讲很有信息量，但旁听二十多位受过高等教育的工程师争论如何敲定最后细节让它真正落地，更有教育意义。让我意外的是，我竟然能在一些会议中提供有用的反馈。我在 FreeBSD 圈外的许多活动上做演讲，能分享不少在 Linux 会议上的见闻。一个对经验丰富的开发者好用的工具，未必对所有人都好用。

每年 BSDCan 结束时我都疲惫不堪。BSDCam 结束时我已彻底精疲力尽。这些高强度、聚焦明确的讨论让我的大脑被拉伸到极限。不过，我对 FreeBSD 开发关键部分的真实走向有了非常清晰的了解。这有助于让新版《Absolute FreeBSD》经得起未来考验——至少在计算机书能经得起未来考验的范围内。此外，BSDCam 的会议晚宴是我见过最华丽的。我要感谢 Robert Watson 的盛情邀请，并感谢 FreeBSD 基金会为本趟行程分担费用。
